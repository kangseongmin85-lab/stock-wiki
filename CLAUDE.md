# LLM Wiki — 주식 투자 지식 베이스 운영 가이드

> Notion "종목재료정리" DB를 Raw Source로 활용하는 LLM 기반 투자 위키 시스템

---

## 세션 부팅 절차 (매 대화 시작 시 반드시 실행)

> ⚠️ **모든 작업 전 `RULES.md` 를 먼저 읽는다.** 절대 누락 불가 규칙이 정리되어 있다.

LLM은 새 대화가 시작되면 아래 순서대로 위키를 읽고 현재 상태를 파악한 뒤 요청을 처리한다.

```
0. RULES.md            → 절대 규칙 확인 (매 세션 필수)
1. wiki/index.md       → 전체 위키 지도 파악 (어떤 종목/테마가 있는지)
2. wiki/overview.md    → 현재 시장 국면 파악 (시황, 주도 섹터, 수급 흐름)
3. 요청 관련 파일      → themes/[테마명].md 또는 stocks/[종목명].md
4. wiki/log.md         → 최근 10줄 확인 (이전 세션에서 무슨 작업을 했는지)
```

> 5단계를 모두 읽은 뒤 "위키 로드 완료. 현재 국면: [국면], 최근 작업: [요약]" 형태로 보고하고 요청 처리 시작.

## 핵심 아이디어

지금 Notion의 "종목재료정리" DB에는 종목명, 관련테마, 키워드, 기사 링크, 등락률, 재료 요약이 축적되어 있습니다. 하지만 이 데이터는 현재 **조회용**으로만 쓰이고 있습니다. 새 기사를 볼 때마다 LLM에게 처음부터 다시 설명해야 하고, 테마 간 연결이나 과거 패턴과의 비교는 머릿속에서만 이루어집니다.

여기서 제안하는 방식은 다릅니다.

LLM이 Notion DB를 **원본 소스(Raw Source)로 읽고**, 그 위에 **별도의 투자 위키를 로컬 파일로 직접 구축·유지**합니다. 새 종목 재료가 들어올 때마다 LLM이 해당 테마 페이지, 대장주 분석 페이지, 수급 흐름 페이지를 자동으로 업데이트합니다. 테마에 수급이 몰리는 순간 — **"지금 대장주는 누구고, 어떻게 매매해야 하는가"** — 를 즉시 답할 수 있는 구조가 됩니다.

**위키는 복리로 쌓이는 투자 두뇌입니다.** 재료·차트·거래량·시황의 연결고리는 이미 만들어져 있고, 과거 패턴과의 모순은 이미 표시되어 있습니다. 종목과 테마를 추가할수록 위키는 더 정교해집니다.

## 현재 Notion DB 구조 (Raw Source)

- **DB 이름**: 종목재료정리
- **접근 방식**: 읽기 전용. **절대 수정 금지.**
  - **단 하나의 예외 — 빈 분류 자동충전**: `카테고리·관련테마·키워드` 가 **비어 있는** 종목에 한해, **기존에 저장된 옵션 어휘 안에서만** 골라 채우는 것은 허용. 신규 옵션 생성·기존 값 변경·다른 컬럼 수정은 여전히 금지. (절차는 아래 「빈 분류 자동충전」 참고. `notion_pusher.py` 가 이미 종목명/등락률/기사/링크/최근업데이트를 upsert하고 있으며, 분류 3종은 그동안 비어 있었음.)

주요 컬럼:

| 컬럼 | 내용 |
|------|------|
| **종목명** | 종목 식별자 |
| **관련테마** | 멀티셀렉트 (예: #HBM, #자율주행, #방산 등) |
| **카테고리** | 반도체 / AI·데이터센터 / 바이오 / 방산 등 |
| **키워드** | 핵심 투자 키워드 |
| **등락률 (%)** | 당일 등락률 |
| **오늘 기사 내용** | 당일 주요 기사 본문 및 링크 |
| **종목 재료 요약** | 요약 (LLM이 자동 채울 대상) |
| **링크** | 관련 기사 URL |
| **최근업데이트** | 마지막 업데이트 날짜 |

### 종목 페이지 본문 (종목명 클릭 시)

각 종목의 Notion 페이지 안에는 DB 컬럼 외에 아래 본문 정보가 추가로 포함되어 있다. Ingest 시 반드시 페이지 본문까지 읽어야 한다.

| 항목 | 내용 |
|------|------|
| **홈페이지** | 기업 공식 홈페이지 URL |
| **재무사항** | 매출액, 영업이익, 영업이익률, 순이익, 부채비율, PER, PBR, ROE, 시가총액 등 |
| **추가 중요내용** | 사업 구조, 주요 고객사, 경쟁사, 리스크 요인, 기타 투자 참고 메모 |

> Notion MCP로 페이지 본문 접근 시: DB 컬럼 읽기와 별도로 `notion-fetch` 또는 페이지 ID 직접 조회로 본문 블록을 읽어온다.

## 위키 구조 (로컬 파일)

섹터와 테마는 `themes/` 폴더로 통합. frontmatter의 `category` 필드로 구분.

```
wiki/
├── index.md               # 전체 페이지 목록 + 한줄 요약 (세션 부팅 시 1순위)
├── log.md                 # 시간순 활동 기록 (세션 부팅 시 4순위)
├── overview.md            # 시황, 주도 섹터, 수급 흐름 (세션 부팅 시 2순위)
│
├── themes/                # 테마 + 섹터 통합 관리
│   ├── HBM.md             # category: theme
│   ├── 방산.md            # category: theme
│   ├── 자율주행.md        # category: theme
│   ├── 반도체.md          # category: sector
│   ├── 바이오.md          # category: sector
│   └── _TEMPLATE.md
│
├── stocks/                # 종목별 분석 페이지
│   ├── 삼성전자.md
│   ├── SK하이닉스.md
│   └── _TEMPLATE.md
│
└── analysis/              # 질문·분석으로 생성된 페이지 (저장 트리거 참고)
    └── 스크리닝_YYYY-MM-DD.md
```

### 섹터 vs 테마 구분 기준

| 구분 | 정의 | 예시 | category 값 |
|------|------|------|-------------|
| **섹터** | 산업 분류. 종목들의 넓은 소속 그룹 | 반도체, 바이오, 방산, 화학 | `sector` |
| **테마** | 수급을 끌어당기는 재료 기반 그룹. 섹터를 가로지를 수 있음 | HBM, 자율주행, 휴머노이드 | `theme` |

> 실전 팁: "이 종목이 왜 올랐나?"는 테마 페이지를 보고, "이 업종 전반 흐름은?"은 섹터 페이지를 본다.

## 재료 강도 분류 기준

Ingest 및 스크리닝 시 반드시 아래 기준으로 분류한다.

| 등급 | 정의 | 예시 | 투자 관점 |
|------|------|------|-----------|
| **단기 이슈** | 1~2주 내 소멸 가능한 뉴스성 재료 | 계약 발표, 깜짝 실적, 단발 수주 | 모멘텀 매매. 추격 주의 |
| **중기 모멘텀** | 시황 이슈로 수개월 지속되는 재료 | 전쟁·지정학, 금리 방향 전환, 정책 수혜 | 스윙 매매. 재료 지속성 점검 필요 |
| **구조적 성장** | 산업 패러다임 변화, 다년간 지속 | 반도체 사이클, AI 산업 개화, 고령화 | 중장기 보유. 단기 조정에 흔들리지 않음 |

> 판단이 애매한 경우: 재료가 "뉴스 1개"에서 왔으면 단기, "정책/전쟁/기술 전환"에서 왔으면 중기, "10년 트렌드"와 연결되면 구조적으로 분류.

## 기사 업로드 → 종목 스크리닝 워크플로

### 분석 순서

**Step 1 — 시황 파악**
- 미국 증시 흐름 (나스닥·S&P500·필라델피아 반도체지수)
- 환율·금리·유가 등 매크로 변수
- 코스피·코스닥 전반 수급 방향
- 결론: 공격적 매수 / 선별 매수 / 관망 국면 판단

**Step 2 — 테마 추출**
- 반복 등장하는 키워드 = 현재 시장이 주목하는 테마
- 테마별 기사 밀도(언급 횟수)로 수급 집중도 추정
- `themes/` 페이지와 대조해 신규 테마인지, 기존 테마의 재점화인지 구분

**Step 3 — 종목 스크리닝**

| 기준 | 평가 내용 | 비고 |
|------|-----------|------|
| **재료** | 재료 강도 분류 (위 기준 적용), 재료 신선도, 위키 과거 재료와 비교 | LLM 직접 분석 |
| **거래량** | 기사에서 언급된 거래량 급증 여부, 수급 주체 언급 | 기사 텍스트 기반 |
| **시황** | Step 1 시황과의 정합성 | LLM 직접 분석 |
| **차트** | ⚠️ 직접 분석 불가 → 확인해야 할 패턴 가이드 제공 | 성민님이 직접 확인 |

**Step 4 — 종목 추천 리포트 출력**

```
## 📊 오늘의 시황
[시황 요약 2~3줄]
현재 국면: 선별 매수 / 공격 매수 / 관망

## 🔥 주목 테마 순위
1. [테마명] — 기사 언급 N회 / [신규 테마 or 재점화] / [재료 등급]

## 🎯 매매 후보 종목

### 1순위: [종목명] — [한줄 요약]
- 재료: [내용] / 등급: [단기 이슈 / 중기 모멘텀 / 구조적 성장]
- 거래량: [기사 내 언급 근거]
- 시황 정합성: [현재 시장과 맞는 이유]
- 차트 확인 포인트:
  - [ ] 전고점 돌파 여부 (거래량 동반 필수)
  - [ ] 5일선 / 20일선 정배열 여부
  - [ ] 당일 거래량 20일 평균 대비 N배 이상
  매수 시나리오: 위 조건 중 N개 충족 시
  손절 기준: [지지선] 하향 이탈 시
- 리스크: [주의할 점]

## 📁 위키 업데이트 대상
- 신규 종목 페이지 생성 필요: [종목명]
- 기존 테마 페이지 업데이트 필요: [테마명]
- 신규 테마 페이지 생성 필요: [테마명]
```

## 핵심 운영 방식

### 📥 Ingest (재료 추가 시)

> ⚠️ **Ingest 시작 전 `RULES.md` 규칙 2·3·4 반드시 확인.**

전제조건: Notion MCP 연결 확인 (`종목재료정리` DB 접근 가능 여부 확인)

1. 요청: *"삼성전자 재료 ingest 해줘"*
2. LLM 실행 순서:
   - Notion `종목재료정리` DB에서 해당 종목 DB 컬럼 읽기
   - 해당 종목 Notion 페이지 본문 읽기 (홈페이지, 재무사항, 추가 중요내용, **지난기사 아카이브 + 링크**)
   - 재료 분석 및 강도 분류 (단기 이슈 / 중기 모멘텀 / 구조적 성장)
   - `wiki/stocks/[종목명].md` 업데이트 (재무 섹션 포함)
   - 연관 테마 페이지 업데이트
   - `wiki/log.md` 기록 (overview.md는 밤 9시 스케줄에서 일괄 업데이트)

> 재무사항은 분기 실적 발표 이후에만 변경되므로, 이미 `wiki/stocks/[종목명].md`에 재무 섹션이 존재하면 Notion에서 새 수치가 확인된 경우에만 덮어쓴다. 변경 없으면 건너뜀.

#### ⚠️ Ingest 필수 규칙 (wiki/_errors.md 기반)

**[규칙 1 — 재무 데이터]** Notion 재무사항 표는 wiki에 그대로 옮기지 않는다. `python fetch_finance.py [종목명]` 실행으로 DART OpenAPI에서 자동 수집한다. 미실행 종목은 재무 섹션을 `[⏳ fetch_finance.py 실행 필요]`로 표기한다.

**[규칙 2 — 태그]** Notion 관련테마 멀티셀렉트 값은 필터링 없이 100% 그대로 `tags`, `theme` 필드에 복사한다. ingest 후 Notion 개수 = wiki 개수 반드시 확인.

**[규칙 3 — 재료 타임라인 + 기사 아카이브]** Notion 2개 섹션을 100% 전수 반영한다: ① `추가 중요내용`, ② `지난기사 아카이브 + 링크`. 누락·요약 금지.

타임라인 항목 형식: `- [YYYY-MM-DD] [재료 내용] / [등락률: +X%] / [단기/중기/구조적] / [기사 제목](URL)`
- **등락률**: Notion DB `등락률(%)` 컬럼 값을 해당 날짜 항목에 기재. 없으면 `등락률 미기재` 표기
- **링크**: Notion `링크` 컬럼 및 페이지 본문 URL 100% 보존. 없으면 링크 항목 생략
- **기사 아카이브**: `지난기사 아카이브 + 링크` 섹션은 wiki `## 기사 아카이브`에 전문 그대로 복사 (요약·생략 금지)
- LLM 자체 생성 항목 금지. 출처 불명 항목은 `[출처 미확인]` 표기 후 사용자 확인 요청

#### ✅ Ingest 완료 후 체크리스트

- [ ] 재무 섹션: `python fetch_finance.py [종목명]` 실행 여부 확인 (미실행 시 `[⏳ fetch_finance.py 실행 필요]` 표기)
- [ ] 태그 수: Notion 관련테마 N개 = wiki tags+theme N개 일치 확인
- [ ] 타임라인: 각 항목에 등락률 + 링크 포함 확인 (등락률 없으면 `등락률 미기재`, 링크 없으면 생략)
- [ ] 기사 아카이브: Notion `지난기사 아카이브 + 링크` 100% 전수 복사 확인 (누락·요약 금지)
- [ ] log.md 기록 완료

주 1회 또는 장 마감 후:
- 서로 모순되는 종목 투자 포인트 없는지
- [stale] 태그 달린 오래된 재료/전망 없는지
- 새로 부각되는 테마인데 위키 페이지 없는지
- Notion DB에 새로 추가됐는데 위키에 반영 안 된 종목 없는지

## overview.md 업데이트 스케줄

- **업데이트 주기**: 주중(월~금) 밤 9시 (자동 스케줄 `wiki-ingest-weekday-9pm`)
- **업데이트 내용 (5단계 순서)**:
  1. Python ingest 스크립트 실행 (`--no-finance --delete-orphans --enrich`)
  2. 시장 데이터 수집 (WebSearch/WebFetch):
     - 코스피/코스닥 종가 및 등락률
     - 코스피/코스닥 상승·하락·보합 종목 수
     - 기관·외국인·개인 순매수 금액 (억원, 코스피 기준)
     - 나스닥 선물 / 코스피200 야간선물 / 코스닥150 야간선물 등락률
  3. Notion `종목재료정리` DB에서 당일 업데이트 종목 전체 읽기 (테마별 그루핑)
  4. `wiki/overview.md` 업데이트 (지수·수급·선물·활성테마·국면 판단 포함)
  5. `wiki/log.md` 기록 + 사용자에게 요약 보고
- **개별 ingest 시**: overview.md는 건드리지 않음. 밤 9시 스케줄에서 일괄 반영.

## 테마 시세·순환테마 트래킹 (네이버 금융 자동수집)

네이버 금융 테마별 시세(`https://finance.naver.com/sise/theme.naver`)를 수집해 강세 테마·거래대금·순환(로테이션)을 분석하는 파이프라인.

| 스크립트 | 역할 | 산출물 |
|----------|------|--------|
| `fetch_themes.py` | 265개 테마 + 구성종목(현재가·등락률·거래량·**거래대금**·편입사유) 수집 | `data/themes/날짜_themes.csv`, `날짜_stocks.csv`, `wiki/analysis/시황_날짜.md` (강세순+거래대금순+위키 주목종목⭐ 매칭) |
| `track_rotation.py` | 누적 csv 비교 → 가열/냉각/신규 로테이션 분석 | `wiki/analysis/순환테마_트래킹.md` (최신), `순환테마_날짜.md` |

- **⭐ 주목종목 기준**: 최근 14일 내 `last_updated`된 `stocks/` 종목. (위키가 전 상장사 2300여개를 커버하므로 "최근 재료 들어온 종목"만 신호로 사용)
- **배치 연동**: `리포트_실행.bat`이 `fetch_themes.py`를 백그라운드 동시 시작(매일 수집·누적). `시황분석_실행.bat`이 `track_rotation.py` 실행(누적분으로 순환 분석 갱신).
- **수동 트리거**: "오늘 시황 보여줘 / 순환테마 보여줘" 요청 시 → `python fetch_themes.py` 후 `python track_rotation.py` 실행하고 결과 요약.

## 장중 가격 트리거 알림 (price_alert.py)

뉴스봇과 **분리된 전용 텔레그램 봇**(`.env`의 `ALERT_BOT_TOKEN`/`ALERT_CHAT_ID`)으로 장중 가격 신호를 알림.

| 대상 | 출처 | 신호 |
|------|------|------|
| 관심종목 (진입) | `오늘의 관심종목/` 최근 14일 CSV 누적 (최대 120종목) | 🚀 60일 전고점 돌파(거래량 1.5배 동반), 🔥 거래량 20일평균 대비 3배 급증 |
| 보유종목 (리스크) | `보유종목.csv` (종목명,종목코드,평단가,손절선,메모) | ⛔ 손절선 이탈, 📉 20일선 신규 하향 이탈, 🩸 당일 -5% 급락 |

- **실행 (기본 = 클라우드)**: GitHub Actions `price_alert.yml` — 30분마다 깨어나 28분간 **내부 1분 주기 스캔** (시세 배치 조회 5회/스캔). 알림 지연 최대 ~70초. **PC 꺼져 있어도 동작**. 로컬 `가격알림_실행.bat`은 백업/테스트용
- **감시목록 동기화**: 공개 repo라 CSV를 올리지 않음 — `리포트_실행.bat` 마지막 단계가 `sync_watchlist_github.py`(관심종목 → `ALERT_WATCHLIST` Secret) + `gh secret set ALERT_HOLDINGS < 보유종목.csv` 자동 실행. **저녁 리포트를 돌려야 다음날 클라우드 감시목록이 갱신됨**
- **Secrets (등록 완료)**: `ALERT_BOT_TOKEN`, `ALERT_CHAT_ID`, `ALERT_WATCHLIST`, `ALERT_HOLDINGS`
- **데이터**: 기준선(60일 고점·20일 평균거래량·20일선)은 네이버 일봉에서 하루 1회 캐시, 실시간가는 네이버 폴링 API. 클라우드에선 actions/cache로 run 간 상태 유지
- **dedup**: 같은 날 같은 종목·조건 1회만 (`_cache/price_alert_sent.json`)
- **클라우드 수동 실행**: `gh workflow run price_alert.yml -R kangseongmin85-lab/stock-wiki -f mode=test`(연결점검) / `mode=force`(장외 강제 스캔)
- **로컬 테스트**: `python price_alert.py --dry-run --force`

## 빈 분류 자동충전 (taxonomy_backfill.py)

리포트 푸시(`notion_pusher.upsert_stock`)는 `카테고리·관련테마·키워드` 를 채우지 않아 신규/일부 종목은 이 3종이 비어 있다. 이를 **이미 DB에 저장된 옵션 어휘 안에서만** 골라 채운다. **분류 판단은 Claude(나)가 세션에서 직접 수행** — 별도 LLM API 미사용.

| 컬럼 | 채우는 값 출처 |
|------|----------------|
| **카테고리** (33개 옵션) | `카테고리` 에 저장된 단어 중 종목 사업분류에 맞는 것 |
| **관련테마** (338개 옵션) | `관련테마` 에 저장된 단어 중 사업정체성+재료드라이버에 맞는 것 |
| **키워드** (336개 옵션) | `키워드` 에 저장된 단어 (관례상 관련테마와 미러, `#` 접두사만 차이) |

> 새 단어 생성 금지. 어휘에 없는 값은 스크립트가 구조적으로 거부한다. 이미 값이 있는 컬럼은 절대 덮어쓰지 않는다(빈 칸만).

**스크립트 역할 (탐지 + 검증 기록만, 판단은 Claude):**
- `python taxonomy_backfill.py --list --today --out reports\taxonomy_todo.json` — 오늘 푸시분 중 빈 종목 + 분류용 텍스트(사업요약·오늘기사) + 현재 옵션 어휘 덤프. (`리포트_실행.bat` [1/4] 푸시 직후 자동 실행)
- `python taxonomy_backfill.py --list` — 전체 빈 종목 / `--names A,B` 특정 종목 / `--options` 어휘만 갱신
- `python taxonomy_backfill.py --apply picks.json` — Claude가 고른 분류 기록(빈 칸만, 어휘 검증). `--dry-run` 미리보기.
  - picks 형식: `[{"name":"HPSP","category":["반도체"],"themes":["#반도체 장비","#온디바이스AI"],"keywords":["반도체 장비","온디바이스AI"]}]`

**Claude 루틴 (요청: "빈 분류 채워줘" 또는 리포트 푸시 직후):**
1. `taxonomy_backfill.py --list --today` (또는 `--list`) 로 빈 종목 + 텍스트 + 어휘 확보
2. 종목별로 사업정체성·재료드라이버 분석 → 각 컬럼 어휘에서 적합값 선택 (판단 애매하면 해당 종목 Notion 페이지 본문을 `notion-fetch` 로 추가 조회)
3. picks.json 작성 → `--apply` (먼저 `--dry-run` 권장)
4. 결과 보고. (위키는 다음 ingest 때 Notion→frontmatter 자동 전파)

> 분류 기준: 사업분류=카테고리, 사업정체성(예: 반도체 장비)+최근 재료드라이버(예: 온디바이스AI)=관련테마, 키워드는 관련테마 미러. 동종 피어 태깅과 일관성 유지(예: 원익IPS=반도체/#반도체 장비). 테마편승·과도태깅 금지 — 사업 실체에 근거.

## 주식 일정 자동 업데이트 (schedule_scan.py + schedule_push.py)

노션 **🗓️ 주식 일정** DB(`database_id=20dffbf4617380a99df6c90c5de6fdc9`, 데이터소스 `collection://20dffbf4-6173-80b1-9dc6-000bdae4c938`)에 **매매에 영향 주는 일정**을 매일 쌓고, 동시에 `wiki/일정.md`(옵시디언 미러)를 자동 동기화한다. taxonomy와 동일한 **"배치 후보덤프 → Claude 적용"** 패턴. **판단은 Claude(나)가 세션에서 직접 수행.**

DB 스키마: `이름`(title) / `날짜`(date, 범위 가능) / `태그`(multi_select).
태그 어휘(4범위): **실적발표 · IPO/상장 · 락업해제 · 거시/정책 · 지정학 · 종목공시**.

| 스크립트 | 역할 | 트리거 |
|----------|------|--------|
| `schedule_scan.py` | (읽기전용) 노션 기존일정(중복기준) + 오늘의 시황 docx 헤드라인 + 네이버 뉴스API(일정 키워드)에서 **날짜 있는 미래 이벤트 후보** 수집 → `reports/schedule_todo.json` | `리포트_실행.bat` [일정] 단계 자동 |
| `schedule_push.py` | Claude가 고른 `picks.json` 을 노션 DB에 upsert(이름+날짜 중복 스킵) + 노션 현재상태로 `wiki/일정.md` 재생성. `--dry-run`/`--sync-wiki` 지원 | Claude가 호출 |

**Claude 루틴 (요청: "일정 업데이트해줘" 또는 리포트 실행 직후):**
1. `reports/schedule_todo.json` 읽기 (없으면 `python schedule_scan.py` 먼저).
2. `candidates` 중 **날짜가 확정된 매매영향 이벤트만** 선별 → 4범위 태그 부여 → `existing_events` 와 중복제거(이름+날짜). 날짜 애매·과거·단순 회고성은 버린다. date_hints가 비었어도 본문에 날짜 단서가 있으면 추론, 불명확하면 제외(억지 추가 금지 — [[taxonomy-fill-no-forcing]] 원칙 동일).
3. `picks.json` 작성: `[{"name","date":"YYYY-MM-DD","end":"(범위면)","tags":[...],"note":""}]`.
4. `python schedule_push.py --apply picks.json --dry-run` (생성/스킵·`reports/일정_preview.md` 확인) → 이상 없으면 `--dry-run` 빼고 실제 실행.
5. 결과 보고 (생성 N · 스킵 N · 다가오는 일정 요약).

> 위키만 다시 맞추려면 `python schedule_push.py --sync-wiki` (노션→`wiki/일정.md`). 시황분석 시 임박 일정은 `wiki/일정.md` 또는 `AIDC_카탈리스트_캘린더.md` 참고.

|------|------|------|
| **IMI 원칙 플레이북** | `wiki/원칙_IMI.md` | 노션 `주식 > IMI` 강의 DB(`collection://2caffbf4-...`)에서 정제한 매매 원칙(진입/지지·저항/분할·비중/손절/현금/수급/시황/심리). 원칙 ID(예: `손절-1`)로 인용. "IMI 원칙 업데이트해줘" 시 재동기화 |
| **노션 매매일지 DB** | `collection://05c63abb-2ddf-490a-8612-e68ee48130e3` (페이지 `2a2867949fc74b2f93a4bc6b82486b8d`, 주식 페이지 하위) | 실제 매매 기록. **속성은 `종목`+`날짜`만**(단순화 — 표는 매매 목록, 클릭하면 상세). 매수/매도가·손익률·매매유형·사유·IMI분석·차트는 **페이지 본문**에 `매매기록` 스킬이 고정 템플릿으로 작성(매매회고가 본문 파싱). 차트 이미지는 페이지에 직접 첨부(자동 업로드 불가) |

## 하네스: 시황 분석

**목표:** 시황 docx + 위키 백데이터 + **실제 가격 레벨** + **IMI 학습 원칙** 결합 → 장 시작 전 한눈에 보는 **⚡오늘의 한 장(정보 요약)**. **처방 아닌 정보 제시** — 팩트를 깔끔히 보여주고 **매매 판단은 성민님이** 한다.

**트리거:** "시황 분석해줘", docx 첨부, "오늘 대비", "매매 국면", "종목 시나리오" 등 시황 분석 관련 요청 시 `시황분석` 스킬을 사용하라. 단순 사실 확인은 직접 응답 가능.

**핵심 산출:** ⚡한 장(국면=시장신호 / 자금 몰리는 곳 / 주목종목 현황 [스윙·단타후보]+레벨·과열 / 보유 현황 / 주의신호 / 관련 IMI 원칙 태그). 레벨은 `python fetch_levels.py --all`(또는 `--names`)가 만든 `_cache/levels_날짜.json`(현재가·5/20/60일선·60일전고·20일이격·연속상승·거래량배수·RR참고). 과열은 형용사 아닌 **숫자**(20일이격≥+25% 등). "사라/팔아라" 단정 금지.
